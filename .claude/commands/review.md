# PR レビュー

このPRを以下の5つの視点から徹底的にレビューしてください。

## 前提

- AI支援開発（Claude Code等）で生成・修正されたコードが含まれる前提でレビューすること
- 各視点から独立して評価し、重複する指摘も視点ごとに明記すること
- 参考基準: [Google Engineering Practices](https://google.github.io/eng-practices/review/)

## 手順

1. `gh pr view --json body` でPR説明文を取得し、**事前確認**を実施する
2. 事前確認でCriticalがあれば、その時点で報告してレビューを中断する
3. `git diff main...HEAD` で現在のブランチの変更差分を確認する
4. 変更ファイル一覧と変更概要を把握する
5. 以下の5視点で順番にレビューを実施する
6. 最後に総合評価を出力する

---

## 事前確認: 📋 PRテンプレート準拠

**目的**: PRの説明がプロジェクトのテンプレート（`.github/pull_request_template.md`）に従って正しく記入されているか確認する

**判定基準**: 未記入・コメントのみ・デフォルト値のままの項目は 🔴 Critical として扱い、コードレビューを開始する前に報告する

**確認項目**:
- [ ] **概要**: テンプレートコメントが残ったままでなく、変更の目的・背景が記述されているか
- [ ] **変更内容**: `-` のプレースホルダーのままでなく、変更点が1つ以上リストアップされているか
- [ ] **関連Issue**: 記入されているか（Issueがない場合は「なし」等の明示があるか）
- [ ] **動作確認**: 全チェックボックスが適切にチェックされているか（未チェックのまま放置されていないか）
- [ ] **レビュー観点**: 空欄でなく、レビュアーへの案内が記載されているか

---

## 視点1: 👔 プロダクトオーナー視点

**目的**: 変更がビジネス価値・要件と整合しているか確認する

**確認項目**:
- [ ] PRの説明が変更の目的・背景を明確に説明しているか
- [ ] 実装がIssue/要件の意図を正確に実現しているか（AIがビジネス意図を誤解していないか）
- [ ] スコープクリープ（要求外の変更・機能追加）がないか
- [ ] ユーザーへの影響（動作・UX変化）が明確か
- [ ] 変更に伴うドキュメント・READMEの更新が必要か
- [ ] 将来の要件変更に対して過度に依存した設計になっていないか

---

## 視点2: 🧪 品質保証 (QA) 視点

**目的**: テスト品質・ロジック正確性・リグレッションリスクを確認する

**確認項目**:
- [ ] 変更に対応するテスト（ユニット/統合）が追加・更新されているか
- [ ] エッジケース（境界値・null・空配列・タイムアウト等）がカバーされているか
- [ ] AIが生成したロジックに「それらしいが間違った」実装がないか（ハルシネーション検出）
- [ ] 既存テストへの影響（壊れるテストがないか）を確認したか
- [ ] 異常系・エラーパスのテストが存在するか
- [ ] 手動テストが必要な箇所が明示されているか
- [ ] CI/CDが正常に通る見込みがあるか

---

## 視点3: 💻 開発者視点

**目的**: コード品質・設計・保守性を確認する（Google Engineering Practices準拠）

**確認項目**:
- [ ] **設計**: アーキテクチャ・モジュール構成が適切か（過剰な抽象化・AIによる不要コードがないか）
- [ ] **複雑性**: 関数・クラスが単一責任を持ち、レビュアーが理解できる複雑さか
- [ ] **命名**: 変数・関数・クラス名が意図を明確に表しているか
- [ ] **コメント**: 「何を」でなく「なぜ」を説明するコメントが書かれているか
- [ ] **DRY**: コードの重複がないか
- [ ] **エラーハンドリング**: 外部API・DBへの呼び出しに適切な例外処理があるか
- [ ] **パフォーマンス**: N+1問題・不要なループ・過剰なAPI呼び出しがないか
- [ ] **コミットメッセージ**: プロジェクト規約（`<type>: <日本語メッセージ>`形式）に従っているか

---

## 視点4: 🔒 セキュリティ視点

**目的**: AI生成コード特有のセキュリティリスクを含む脆弱性を確認する

参考:
- [Vibe Coding Security - Checkmarx](https://checkmarx.com/blog/security-in-vibe-coding/)
- [AI-Generated Code Vulnerabilities - Vidoc Security Lab](https://blog.vidocsecurity.com/blog/vibe-coding-security-vulnerabilities)

**確認項目**:
- [ ] **シークレット**: APIキー・パスワード・トークンがコード・設定ファイルにハードコードされていないか
- [ ] **認証**: 認証ロジックがAIの「それらしい実装」でなく、正しく機能するか
- [ ] **認可 (BOLA)**: リソースIDで参照する処理で、呼び出し元の所有権確認があるか
- [ ] **インジェクション**: SQLクエリ・OSコマンド・テンプレートに文字列結合を使っていないか
- [ ] **入力検証**: 外部入力（Gmail・API等）すべてにバリデーションがあるか
- [ ] **依存パッケージ**: 新規追加パッケージが実在し、公式レジストリから取得しているか（ハルシネーションパッケージ確認）
- [ ] **OAuth連携スコープ**: Gmail等の連携スコープが最小権限になっているか
- [ ] **エラーレスポンス**: スタックトレース・内部情報がユーザーに露出しないか

---

## 視点5: 📦 依存・保守視点

**目的**: 依存関係の健全性と長期保守性を確認する

参考: [mgreiler/code-review-checklist](https://github.com/mgreiler/code-review-checklist)

**確認項目**:
- [ ] 新規依存パッケージの追加に正当な理由があるか
- [ ] ロックファイル（package-lock.json等）が更新・コミットされているか
- [ ] 追加パッケージに既知の脆弱性がないか（npm audit / pip audit等）
- [ ] 使用していない依存が追加されていないか
- [ ] 変更が既存の他コンポーネントに破壊的影響を与えていないか
- [ ] 設定変更（環境変数追加等）がドキュメントに反映されているか

---

## 重要度レベル

| レベル | 意味 | 対応 |
|---|---|---|
| 🔴 **Critical** | マージ前に必ず修正が必要 | ブロック |
| 🟡 **Warning** | 修正を強く推奨（理由があれば許容） | 要議論 |
| 🟢 **Suggestion** | 改善提案（任意対応） | 任意 |
| 💡 **Note** | 情報共有・補足 | 対応不要 |

---

## 出力フォーマット

各視点のレビュー結果を以下の形式で出力すること:

```
### [視点名] レビュー

#### [重要度アイコン] [指摘タイトル]
**ファイル**: `path/to/file` (L行番号)
**指摘**: 具体的な問題点
**提案**: 改善案または推奨アクション
```

指摘がない視点は「✅ 問題なし」と記載すること。

---

## 総合評価

すべての視点のレビュー完了後、以下を出力すること:

**判定**: `✅ Approve` / `🔄 Request Changes` / `💬 Comment`

**サマリー**:
- 🔴 Critical: X件
- 🟡 Warning: X件
- 🟢 Suggestion: X件

**優先対応項目**: （Criticalがある場合のみ箇条書きで列挙）
