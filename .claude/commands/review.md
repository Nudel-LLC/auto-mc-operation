# PR レビュー

このPRをレビューしてください。**コードと設計ドキュメントの双方** に対応するモード型ワークフローです。

---

## 引数

| 引数 | 動作 |
|------|------|
| `/review` | **自動判定モード**(変更ファイルから code / design を判定) |
| `/review code` | コードレビューモード(明示) |
| `/review design` | 設計ドキュメントレビューモード(明示) |
| `/review --since=<comment-id>` | 指定コメント ID 以降の差分のみ再レビュー(指摘の解決状況を中心に確認) |
| `/review <自由記述>` | 視点・ファイル指定など(例: `/review design セキュリティ視点だけ`) |

複数引数の併用可。例: `/review design --since=4365084235`

---

## 前提

- AI 支援開発(Claude Code 等)で生成・修正された成果物が含まれる前提でレビューする
- 各視点から独立して評価し、重複する指摘も視点ごとに明記する
- 参考基準: [Google Engineering Practices](https://google.github.io/eng-practices/review/)

---

## 手順

1. `gh pr view --json body` で PR 説明文を取得し、**事前確認**を実施
2. 事前確認で Critical があれば、その時点で報告してレビューを中断
3. **モード自動判定**(引数で明示されていない場合)
4. `git diff main...HEAD` で変更差分を確認
5. **モード別の 5 視点** で順番にレビューを実施
6. **重要度判定基準マトリクス** に従ってラベリング
7. 最後に総合評価を出力
8. **再レビュー(`--since`)モード** の場合は手順 5 を「前回指摘の解決状況確認」に置換

---

## モード自動判定アルゴリズム

`git diff main...HEAD --name-only` で変更ファイル一覧を取得し、以下の優先順位で判定:

1. 以下のいずれかに該当するファイルがあれば **code モード**
   - 実装言語ファイル: `.rs` / `.ts` / `.tsx` / `.js` / `.jsx` / `.py` / `.go` / `.java` / `.rb` / `.kt` / `.swift` / `.cs` / `.cpp` / `.c` / `.h` / `.scala` / `.php`
   - 依存マニフェスト: `package.json` / `Cargo.toml` / `go.mod` / `pyproject.toml` / `Gemfile` / `pom.xml` / `requirements.txt`
   - ロックファイル: `package-lock.json` / `Cargo.lock` / `poetry.lock` / `Gemfile.lock` / `yarn.lock` / `pnpm-lock.yaml`
   - CI 設定: `.github/workflows/`
   - インフラ定義: `*.tf` / `*.yaml`(k8s)/ `Dockerfile`
2. 上記が **無く**、変更が `*.md` のみ、または `aidlc-docs/` / `docs/` 配下のみ → **design モード**
3. **両方含む** → code モード(設計整合性は code モード PO 視点でカバー)

判定結果は最初に明示する(例: 「自動判定: design モード — 変更は aidlc-docs/inception/ 配下の Markdown のみ」)。

## 事前確認: 📋 PR テンプレート準拠

**目的**: PR の説明がプロジェクトのテンプレート(`.github/pull_request_template.md`)に従って記入されているか確認する。

**判定基準**: 未記入・コメントのみ・デフォルト値のままの項目は重要度判定基準マトリクス(後述)に従って判定。

### PR 種別ごとの確認項目

PR テンプレート冒頭の **「PR 種別」** セクションで選択された種別(`Code` / `Docs` / `AIDLC stage` の 3 種)に応じて確認基準を変える。種別未指定の場合はデフォルトで **Code** とみなす。

| 確認項目 | Code | Docs | AIDLC stage |
|---------|------|------|-------------|
| **概要**(目的・背景) | 必須 🔴 | 必須 🔴 | 必須 🔴 |
| **変更内容**(箇条書き) | 必須 🔴 | 必須 🔴 | 必須 🔴 |
| **関連 Issue / PR / コメント** | 推奨 🟡 | 推奨 🟡 | 推奨 🟡 |
| **動作確認**(ローカル動作・既存影響) | 必須 🔴 | N/A(成果物がドキュメントのみ) | 緩和(代わりに「上流成果物との整合性確認」を必須) |
| **上流成果物との整合性確認** | N/A | N/A | 必須 🔴(AIDLC stage) |
| **レビュー観点** | 推奨 🟡 | 推奨 🟡 | 推奨 🟡 |

Critical が 1 件でもある場合、コードレビューを中断し PR 説明の修正を促す。

## code モード: 5 視点

### 👔 視点 1: プロダクトオーナー視点

**目的**: 変更がビジネス価値・要件と整合しているか確認する。

**確認項目**:
- [ ] PR の説明が変更の目的・背景を明確に説明しているか
- [ ] 実装が Issue / 要件の意図を正確に実現しているか(AI がビジネス意図を誤解していないか)
- [ ] スコープクリープ(要求外の変更・機能追加)がないか
- [ ] ユーザーへの影響(動作・UX 変化)が明確か
- [ ] 変更に伴うドキュメント・README の更新が必要か
- [ ] **段階間整合性**(AIDLC PR の場合): 上流成果物(`requirements.md` / `stories.md` / `application-design.md` 等)に逆波及更新が必要か
- [ ] 将来の要件変更に対して過度に依存した設計になっていないか

### 🧪 視点 2: 品質保証 (QA) 視点

**目的**: テスト品質・ロジック正確性・リグレッションリスクを確認する。

**確認項目**:
- [ ] 変更に対応するテスト(ユニット/統合)が追加・更新されているか
- [ ] エッジケース(境界値・null・空配列・タイムアウト等)がカバーされているか
- [ ] AI が生成したロジックに「それらしいが間違った」実装がないか(ハルシネーション検出)
- [ ] 既存テストへの影響(壊れるテストがないか)を確認したか
- [ ] 異常系・エラーパスのテストが存在するか
- [ ] 手動テストが必要な箇所が明示されているか
- [ ] CI/CD が正常に通る見込みがあるか

### 💻 視点 3: 開発者視点

**目的**: コード品質・設計・保守性を確認する(Google Engineering Practices 準拠)。

**確認項目**:
- [ ] **設計**: アーキテクチャ・モジュール構成が適切か(過剰な抽象化・AI による不要コードがないか)
- [ ] **複雑性**: 関数・クラスが単一責任を持ち、レビュアーが理解できる複雑さか
- [ ] **命名**: 変数・関数・クラス名が意図を明確に表しているか
- [ ] **コメント**: 「何を」でなく「なぜ」を説明するコメントが書かれているか
- [ ] **DRY**: コードの重複がないか
- [ ] **エラーハンドリング**: 外部 API・DB への呼び出しに適切な例外処理があるか
- [ ] **パフォーマンス**: N+1 問題・不要なループ・過剰な API 呼び出しがないか
- [ ] **コミットメッセージ**: プロジェクト規約(`<type>: <日本語メッセージ>`形式)に従っているか

### 🔒 視点 4: セキュリティ視点

**目的**: AI 生成コード特有のセキュリティリスクを含む脆弱性を確認する。

参考: [Vibe Coding Security - Checkmarx](https://checkmarx.com/blog/security-in-vibe-coding/) / [AI-Generated Code Vulnerabilities - Vidoc Security Lab](https://blog.vidocsecurity.com/blog/vibe-coding-security-vulnerabilities)

**確認項目**:
- [ ] **シークレット**: API キー・パスワード・トークンがコード・設定ファイルにハードコードされていないか
- [ ] **認証**: 認証ロジックが AI の「それらしい実装」でなく、正しく機能するか
- [ ] **認可 (BOLA)**: リソース ID で参照する処理で、呼び出し元の所有権確認があるか
- [ ] **インジェクション**: SQL クエリ・OS コマンド・テンプレートに文字列結合を使っていないか
- [ ] **入力検証**: 外部入力(Gmail・API 等)すべてにバリデーションがあるか
- [ ] **依存パッケージ**: 新規追加パッケージが実在し、公式レジストリから取得しているか(ハルシネーションパッケージ確認)
- [ ] **OAuth 連携スコープ**: Gmail 等の連携スコープが最小権限になっているか
- [ ] **エラーレスポンス**: スタックトレース・内部情報がユーザーに露出しないか

### 📦 視点 5: 依存・保守視点

**目的**: 依存関係の健全性と長期保守性を確認する。

**確認項目**:
- [ ] 新規依存パッケージの追加に正当な理由があるか
- [ ] ロックファイル(package-lock.json 等)が更新・コミットされているか
- [ ] 追加パッケージに既知の脆弱性がないか(npm audit / pip audit / cargo audit 等)
- [ ] 使用していない依存が追加されていないか
- [ ] 変更が既存の他コンポーネントに破壊的影響を与えていないか
- [ ] 設定変更(環境変数追加等)がドキュメントに反映されているか

## design モード: 5 視点

設計ドキュメント(`*.md`、AIDLC 成果物等)向け。コードを直接対象としない。

### 🧭 視点 1: 要件整合性視点(段階間整合性含む)

**目的**: 上流成果物と本 PR の成果物の整合性を確認する。

**確認項目**:
- [ ] PR の成果物が上流ドキュメント(`requirements.md` / `stories.md` / `personas.md` 等)の意図と一致しているか
- [ ] **段階間整合性**: 本 PR で確定した判断により、上流ドキュメントへの **逆波及更新** が必要か(例: Application Design で Phase 2 移行を決定 → `requirements.md` の MVP スコープ更新が必須)
- [ ] スコープクリープ(要求外の設計拡張)がないか
- [ ] AI が要件・ペルソナの意図を誤解した設計になっていないか
- [ ] 用語・命名が上流ドキュメントと一貫しているか

### 🧩 視点 2: 設計コヒーレンス視点(内部矛盾)

**目的**: 同 PR 内・同ファイル内・同セクション内の **内部矛盾** を検出する。

**確認項目**:
- [ ] 同じ事象を複数箇所で説明している場合、事実が一致しているか(例: シーケンス図と Queue 構成表の経路一致)
- [ ] 設計判断(Plan の Q&A 回答)と実装方針が一致しているか
- [ ] 章節間の参照が崩れていないか(例: 「セクション 4 参照」とあるがセクション 4 が無い)
- [ ] 表記揺れ(同一概念に異なる名前)がないか
- [ ] エラー型・データ型・API スキーマが複数箇所で食い違っていないか

### 🔍 視点 3: ハルシネーション検出視点(設計版)

**目的**: 設計ドキュメント特有のハルシネーション(存在しない参照・数値矛盾)を検出する。

**確認項目**:
- [ ] **数値整合性**: 冒頭サマリ ↔ 表合計 ↔ 各セクション本文の数値が一致するか(例: 「全 51 コンポーネント」の合計が本当に 51 か)
- [ ] **ID 参照整合性**: F-XX / U-XX / Story ID / 型名 / メソッド名が上流成果物に **実在** するか(存在しない F-15 などへの参照を検出)
- [ ] **重複記述の事実一致**: 同じ事象を 2 箇所以上で説明している場合の整合性(例: 表とシーケンス図で経路が違う)
- [ ] **列挙の網羅性**: 「全 N 件」と宣言している箇所で、本文に N 件すべて存在するか(例: 「10 ユースケース」と書いて 9 個しか定義されていない場合は片手落ち)
- [ ] **メソッド・型の参照整合性**: 引数や戻り値で参照している型が他ファイルで定義されているか

### 🔒 視点 4: セキュリティ視点(設計レベル)

**目的**: 設計ドキュメント段階で識別すべきセキュリティ要件・脅威モデルを確認する。

**確認項目**:
- [ ] 認証・認可方針が **すべての公開エンドポイント** に明示されているか
- [ ] 機密データ(トークン・PII)の保管・暗号化方針があるか
- [ ] BOLA / IDOR 防止策がリソース参照系 API で言及されているか
- [ ] 入力検証・出力エスケープの責務がどのレイヤに属するか明示されているか
- [ ] 監査ログ・改竄防止の機械的保証(トリガー・append-only 等)が指定されているか
- [ ] OAuth スコープ・最小権限原則が言及されているか
- [ ] 設計レベルの脅威モデリング(STRIDE 等)が完了基準に含まれているか

### 🏗️ 視点 5: 構築可能性視点(設計版「依存・保守視点」)

**目的**: 後続ステージ(Construction / Functional Design)で本設計が **実装可能** か、未確定事項が後続を阻害しないかを確認する。

**確認項目**:
- [ ] **未確定事項のリスト化**: 後続ステージで詰まりそうな未確定パラメータ(タイムアウト・リトライ回数・Cron 頻度・コスト上限値)が「Functional Design で確定」と明示されているか
- [ ] **拡張性ポート**: 抽象化レイヤが将来の仕様変更(別 SaaS への乗り換え等)を本当に吸収できる設計か
- [ ] **コスト・性能の達成可能性**: NFR(レイテンシ・スループット・コスト目標)達成の **設計目標値の枠組み** が示されているか(詳細値は Functional Design でも、根拠ある上限が示されているか)
- [ ] **依存外部サービスのバージョン・SLA**: 設計時点で前提としている外部サービスの仕様が明示されているか
- [ ] **テスト戦略との接続**: モック境界・PBT 対象・E2E スコープが Application Design で示されているか

## 重要度判定基準マトリクス

### 共通レベル定義

| アイコン | レベル | 対応 |
|---------|--------|------|
| 🔴 | **Critical** | マージ前に必ず修正(ブロック) |
| 🟡 | **Warning** | 修正を強く推奨(理由があれば許容、要議論) |
| 🟢 | **Suggestion** | 改善提案(任意対応) |
| 💡 | **Note** | 情報共有・補足(対応不要) |

### code モード判定例

| 種類 | 🔴 Critical | 🟡 Warning | 🟢 Suggestion |
|------|-----------|-----------|--------------|
| シークレット | API キーのハードコード | `.env.example` 未更新 | コメントの redact ガイド追記 |
| 認証・認可 | BOLA を引き起こすクエリ | 認可粒度が荒い | テスト追加 |
| インジェクション | SQL/OS コマンドへの文字列結合 | エスケープ漏れの可能性 | パラメータバインドの統一 |
| テスト | 必須経路のテスト未追加 | エッジケース未カバー | テスト命名改善 |
| 依存 | ハルシネーションパッケージ参照 | 既知脆弱性のあるバージョン | バージョン固定の方針 |

### design モード判定例

| 種類 | 🔴 Critical | 🟡 Warning | 🟢 Suggestion |
|------|-----------|-----------|--------------|
| 内部矛盾 | 同じ事象が 2 箇所で別事実(実装で必ず齟齬) | 表記揺れ・粒度不一致 | 補足が望ましい |
| ID 参照 | 存在しない Story / 型 を参照(ハルシネーション) | リンク先未生成 | 別表現が好ましい |
| 数値整合性 | サマリ数 ≠ 詳細合計 | 概算と精密値の混在 | 単位明示の不足 |
| 上流不整合 | 要件と矛盾する設計 | 要件記述が陳腐化 | 用語のゆらぎ |
| 完全性 | 必須要素の欠落(API / UseCase が定義されない) | 詳細未記載 | 図示があると親切 |
| セキュリティ | 公開エンドポイントの認証方針未記述 | 暗号化方針が部分的 | 脅威モデル参照リンク |
| 構築可能性 | 後続実装が不能な未確定事項 | 未確定事項が「TBD」のまま放置 | 上限値の暫定示唆 |

## 出力フォーマット

各視点のレビュー結果を以下の形式で出力する:

### 単一箇所の指摘

```
### [視点名] レビュー

#### [重要度アイコン] [指摘タイトル]
**ファイル**: `path/to/file`
**セクション**: `## 4.2 認証`(または `### Story F-12` など、構造的参照)
**行**: L45 / L120-L135(該当する場合のみ)
**指摘**: 具体的な問題点
**提案**: 改善案または推奨アクション
```

### 複数箇所をまたがる整合性指摘

```
#### [重要度アイコン] [指摘タイトル]
**対象箇所**(複数):
- `services.md` セクション 3「Queue 構成」(L82, A-3 行)
- `services.md` シーケンス図 1(L36-49)
- `services.md` シーケンス図 2(L54-72)
**指摘**: 同一事象を 3 箇所で異なる事実として記述している
**提案**: 真の方針を 1 つ確定し、3 箇所を一致させる(推奨案: ...)
```

指摘がない視点は「✅ 問題なし」と記載すること。

## 総合評価

すべての視点のレビュー完了後、以下を出力すること:

**判定**: `✅ Approve` / `🔄 Request Changes` / `💬 Comment`

**サマリー**:
- 🔴 Critical: X 件
- 🟡 Warning: X 件
- 🟢 Suggestion: X 件
- 💡 Note: X 件

**優先対応項目**: (Critical がある場合のみ箇条書きで列挙)

**レビュー反映確認用 ID**: 本レビューコメントの GitHub コメント ID(例: `4365084235`)を末尾に記載。再レビュー時(`/review --since=<id>`)の起点として参照する。

## 再レビュー(`--since=<comment-id>`)モード

修正反映後の差分のみを再レビューするためのモード。指摘 → 修正 → 再レビューのサイクルを効率化する。

### 手順

1. 引数 `--since=<comment-id>` から **起点となる前回レビューコメント** を取得
   - GitHub MCP の `pull_request_read` で `get_comments` / `get_review_comments` から該当コメントを取得
2. 前回レビュー時点のコミット SHA(コメント本文または PR コミット履歴から推定)と HEAD の **差分 commit 一覧** を取得
3. 前回コメントの **全指摘(Critical / Warning / Suggestion / Note)を一覧化**
4. 各指摘について **解決状況を判定**:
   - ✅ 解決(該当箇所が修正されている)
   - ⚠️ 部分解決(一部のみ修正)
   - ❌ 未解決(変化なし)
   - ❓ 確認不能(該当箇所が大幅に変わって直接比較できない)
5. **新規導入された問題** がないかも併せて確認(差分コミットで持ち込まれた regression)
6. 総合評価では「前回 Critical N 件 → 今回 0 件」のような **解決ステータスサマリ** を提示

### 出力フォーマット(再レビュー版)

```
### 🟢 修正確認

#### 🔴 Critical(N 件 / 全解決)

| ID | 検証結果 |
|----|---------|
| **C1** [前回指摘の要約] | ✅ 完全解決。[根拠ファイル/セクション]で確認 |
| ... | ... |

#### 🟡 Warning(M 件 / X 件解決 / Y 件部分解決 / Z 件未解決)
...

### 軽微な所感(対応不要・参考)
- 💡 Note: ...

---

## 総合評価

判定: ✅ Approve(または 🔄 Request Changes)

サマリー: 🔴 0 / 🟡 0 / 🟢 0 / 💡 N
```

## 補足: モードに応じた省略ルール

- **code モード で動作確認の Critical** が PR 種別 = AIDLC stage の場合は緩和(代わりに「上流成果物との整合性確認」を必須化)
- **design モード で QA / 依存・保守視点** は適用せず、代わりに **設計コヒーレンス / ハルシネーション / 構築可能性視点** を実施
- **`--since` 指定時** は事前確認をスキップ可(初回レビューで実施済の前提)
