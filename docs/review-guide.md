# `/review` コマンド 使い方ガイド

## 概要

`/review` は PR の変更内容をレビューする Claude Code スラッシュコマンドです。

**code(実装コード)と design(設計ドキュメント)の 2 モード** を備え、変更内容に応じて自動判定または明示指定でレビュー視点を切り替えます。AI 支援開発(Claude Code 等)で生成された成果物に特有のリスク(ハルシネーション・セキュリティ不備・ビジネス意図の誤解・段階間整合性崩壊)を多角的にレビューします。

## 使い方

### 基本的な使い方

PR ブランチにチェックアウトした状態で、Claude Code のチャットに入力します:

```
/review
```

Claude が以下の順で自動実行します:
1. **モード自動判定**(code / design)
2. **事前確認**: PR テンプレートへの準拠チェック(PR 種別ごとに基準)
3. **モード別レビュー**: 5 視点でレビュー
4. **総合評価**: 判定とサマリーを出力(レビュー反映確認用 ID 付き)

> **注意**: 事前確認で Critical が見つかった場合、コードレビューを中断して PR 説明の修正を求めます。

### モード明示指定

```
/review code      # コードレビューモードを明示
/review design    # 設計ドキュメントレビューモードを明示
```

### 再レビューモード(`--since`)

修正反映後の差分のみを再レビューします:

```
/review --since=4365084235
```

`<comment-id>` には前回のレビューコメントの GitHub ID を指定。Claude は前回指摘 → 解決状況を判定し、未解決・新規問題のみ報告します(全 14 件 → 解決 14 件のような形でサマリ提示)。

### 特定の視点だけ依頼する場合

```
/review セキュリティ視点だけレビューしてください
/review 開発者視点と QA 視点でレビューしてください
/review design ハルシネーション検出視点だけ
```

### 特定ファイルを指定する場合

```
/review src/gmail/client.py のレビューをしてください
/review design aidlc-docs/inception/application-design/ だけ
```

## モード自動判定アルゴリズム

`git diff main...HEAD --name-only` で変更ファイル一覧を取得し、以下の優先順位で判定します:

1. 以下のいずれかに該当するファイルがあれば **code モード**
   - 実装言語ファイル: `.rs` / `.ts` / `.tsx` / `.js` / `.jsx` / `.py` / `.go` / `.java` / `.rb` / `.kt` / `.swift` / `.cs` / `.cpp` / `.c` / `.h` / `.scala` / `.php`
   - 依存マニフェスト: `package.json` / `Cargo.toml` / `go.mod` / `pyproject.toml` / `Gemfile` / `pom.xml` / `requirements.txt`
   - ロックファイル: `package-lock.json` / `Cargo.lock` / `poetry.lock` / `Gemfile.lock` / `yarn.lock` / `pnpm-lock.yaml`
   - CI 設定: `.github/workflows/`
   - インフラ定義: `*.tf` / `*.yaml`(k8s)/ `Dockerfile`
2. 上記が **無く**、変更が `*.md` のみ、または `aidlc-docs/` / `docs/` 配下のみ → **design モード**
3. **両方含む** → code モード(設計整合性は code モード PO 視点でカバー)

判定結果は最初のメッセージで明示されます(例: 「自動判定: design モード — 変更は aidlc-docs/inception/ 配下の Markdown のみ」)。

## 事前確認: PR テンプレート準拠

コードレビューの前に、PR の説明文が `.github/pull_request_template.md` に沿って記入されているかを確認します。

### PR 種別ごとの確認基準

PR テンプレート冒頭の **「PR 種別」** セクションで選択された種別に応じて確認基準を変えます:

| 確認項目 | Code | Docs | AIDLC stage |
|---------|------|------|-------------|
| 概要 | 必須 🔴 | 必須 🔴 | 必須 🔴 |
| 変更内容 | 必須 🔴 | 必須 🔴 | 必須 🔴 |
| 関連 Issue / PR / コメント | 推奨 🟡 | 推奨 🟡 | 推奨 🟡 |
| 動作確認(ローカル動作・既存影響) | 必須 🔴 | N/A | 緩和 |
| **上流成果物との整合性確認** | N/A | N/A | 必須 🔴(AIDLC stage) |
| レビュー観点 | 推奨 🟡 | 推奨 🟡 | 推奨 🟡 |

Critical が 1 件でもある場合、Claude はコードレビューを中断し PR 説明の修正を促します。

> **AIDLC stage PR の特例**: AIDLC ワークフローのステージ完了 PR は本質的に成果物がドキュメントのみであり「ローカル動作確認」が意味をなしません。代わりに **上流成果物(`requirements.md` / `stories.md` 等)との整合性確認** を必須としています。

## code モード: 5 視点

### 👔 プロダクトオーナー視点

ビジネス価値・要件の整合性を確認します。

**主な確認内容**:
- 実装が要件・Issue の意図を正確に実現しているか
- スコープクリープがないか
- ユーザーへの影響が明確か
- AI がビジネス意図を誤解した実装をしていないか
- **段階間整合性(AIDLC PR)**: 上流ドキュメントへの逆波及更新が必要か

### 🧪 品質保証 (QA) 視点

テスト品質とリグレッションリスクを確認します。

**主な確認内容**:
- ユニットテスト・統合テストが追加・更新されているか
- エッジケース(null・境界値・タイムアウト等)がカバーされているか
- **AI ハルシネーション検出**: 「それらしいが間違った」ロジックがないか
- 既存テストへの影響

### 💻 開発者視点

コード品質・設計・保守性を確認します。[Google Engineering Practices](https://google.github.io/eng-practices/review/reviewer/looking-for.html) の 10 カテゴリ準拠。

**主な確認内容**:
- 設計の適切さ(過剰な抽象化・AI による不要コードの排除)
- 単一責任・DRY 原則
- 命名・コメントの質
- エラーハンドリング・パフォーマンス
- コミットメッセージ規約の遵守

### 🔒 セキュリティ視点

AI 生成コード特有のリスクを含むセキュリティ脆弱性を確認します。

> **背景**: Veracode 2025 レポートによると、AI 生成コードは人間のコードより **2.74 倍** 脆弱性が多く、検証した 200 以上の AI 開発アプリの **91.5%** にハルシネーション由来の脆弱性が存在しました。

**主な確認内容**:
- シークレット・API キーのハードコード
- 認証・認可ロジックの正確性(Broken Object-Level Authorization)
- インジェクション(SQL・OS・テンプレート)
- **Gmail 等 OAuth 連携のスコープが最小権限か**
- **依存パッケージのハルシネーション**

### 📦 依存・保守視点

依存関係の健全性と長期保守性を確認します。

**主な確認内容**:
- 新規パッケージ追加の正当性
- ロックファイルの更新状況
- 既知の脆弱性(npm audit / pip audit / cargo audit 等)
- 環境変数追加等の設定変更のドキュメント反映

## design モード: 5 視点

設計ドキュメント(Markdown 成果物・AIDLC 中間生成物)に対するレビュー視点。コードを直接対象としません。

### 🧭 要件整合性視点(段階間整合性含む)

上流成果物(`requirements.md` / `stories.md` / `personas.md` 等)との整合性を確認します。

**主な確認内容**:
- 上流ドキュメントの意図と一致しているか
- **段階間整合性**: 本 PR で確定した判断により上流の **逆波及更新** が必要か
- スコープクリープ(要求外の設計拡張)がないか
- 用語・命名が上流と一貫しているか

> **段階間整合性の例**: Application Design で F8(請求 CSV)を Phase 2 移行と決定した場合、`requirements.md` の MVP スコープ列挙も同期更新が必要。これを怠ると 3 文書間で齟齬が生じる。

### 🧩 設計コヒーレンス視点(内部矛盾)

同 PR 内の内部矛盾を検出します。

**主な確認内容**:
- 同じ事象が複数箇所で別事実として記述されていないか(例: シーケンス図 ↔ Queue 構成表)
- 設計判断(Plan の Q&A 回答)と実装方針が一致しているか
- 章節間の参照が崩れていないか
- 表記揺れ(同一概念に異なる名前)がないか
- エラー型・データ型・API スキーマの食い違い

### 🔍 ハルシネーション検出視点(設計版)

設計ドキュメント特有のハルシネーションを検出します。AI が生成した設計ドキュメントには **「コード上では明らかに分からないが設計レベルでよく出る誤り」** があり、本視点ではそれを系統的に検出します。

**4 つの検出パターン**:

| パターン | 例 | 検出方法 |
|---------|---|---------|
| **数値整合性** | 「全 51 コンポーネント」と書きつつ表合計が 49 | 冒頭サマリ ↔ 表合計 ↔ 各セクションの数値を突合 |
| **ID 参照整合性** | 存在しない Story `F-15` を参照(実在は F-14 まで) | 参照されている全 ID を上流成果物で `grep` 確認 |
| **重複記述の事実一致** | 同じ事象を 3 箇所で異なる事実として記述 | 同じキーワードで複数箇所の記述を比較 |
| **列挙の網羅性** | 「10 ユースケース」と書いて 9 個しか定義されていない | 完了基準で挙げた要素が本文に存在するか確認 |

### 🔒 セキュリティ視点(設計レベル)

設計段階で識別すべき脅威モデル・セキュリティ要件を確認します。

**主な確認内容**:
- 公開エンドポイントすべてに認証・認可方針が明示されているか
- 機密データ(トークン・PII)の保管・暗号化方針があるか
- BOLA / IDOR 防止策がリソース参照系 API で言及されているか
- 監査ログ・改竄防止の機械的保証(トリガー・append-only 等)が指定されているか
- OAuth スコープ・最小権限原則の言及
- 設計レベルの脅威モデリング(STRIDE 等)

### 🏗️ 構築可能性視点

後続ステージ(Construction / Functional Design / Build and Test)で本設計が **実装可能** か、未確定事項が後続を阻害しないかを確認します。

**主な確認内容**:
- **未確定事項のリスト化**: タイムアウト・リトライ回数・コスト上限値などが「Functional Design で確定」と明示されているか
- **拡張性ポート**: 抽象化レイヤが将来の仕様変更を本当に吸収できる設計か
- **コスト・性能の達成可能性**: NFR 達成の **設計目標値の枠組み** が示されているか
- **テスト戦略との接続**: モック境界・PBT 対象・E2E スコープ

> **設計ドキュメントでは「依存・保守視点」が機能しないため、構築可能性視点で代替** します。

## 重要度レベルと判定基準マトリクス

### 共通レベル定義

| アイコン | レベル | 対応方針 |
|---------|--------|---------|
| 🔴 | **Critical** | マージ前に必ず修正。Claude は `Request Changes` 判定 |
| 🟡 | **Warning** | 修正を強く推奨。理由があれば許容 |
| 🟢 | **Suggestion** | 改善提案。対応は任意 |
| 💡 | **Note** | 情報共有・補足。対応不要 |

### code モード判定例

| 種類 | 🔴 Critical | 🟡 Warning | 🟢 Suggestion |
|------|-----------|-----------|--------------|
| シークレット | API キーのハードコード | `.env.example` 未更新 | コメントの redact ガイド追記 |
| 認証・認可 | BOLA を引き起こすクエリ | 認可粒度が荒い | テスト追加 |
| インジェクション | SQL/OS コマンドへの文字列結合 | エスケープ漏れの可能性 | パラメータバインドの統一 |
| テスト | 必須経路のテスト未追加 | エッジケース未カバー | テスト命名改善 |
| 依存 | ハルシネーションパッケージ参照 | 既知脆弱性のあるバージョン | バージョン固定の方針 |

### design モード判定例

| 種類 | 🔴 Critical | 🟡 Warning | 🟢 Suggestion |
|------|-----------|-----------|--------------|
| 内部矛盾 | 同じ事象が 2 箇所で別事実(実装で必ず齟齬) | 表記揺れ・粒度不一致 | 補足が望ましい |
| ID 参照 | 存在しない Story / 型 を参照(ハルシネーション) | リンク先未生成 | 別表現が好ましい |
| 数値整合性 | サマリ数 ≠ 詳細合計 | 概算と精密値の混在 | 単位明示の不足 |
| 上流不整合 | 要件と矛盾する設計 | 要件記述が陳腐化 | 用語のゆらぎ |
| 完全性 | 必須要素の欠落(API/UseCase が未定義) | 詳細未記載 | 図示があると親切 |
| セキュリティ | 公開エンドポイントの認証方針未記述 | 暗号化方針が部分的 | 脅威モデル参照リンク |
| 構築可能性 | 後続実装が不能な未確定事項 | 「TBD」のまま放置 | 上限値の暫定示唆 |

## 出力フォーマット

### 単一箇所の指摘

```
### [視点名] レビュー

#### 🟡 エッジケースのテストが不足しています
**ファイル**: `src/gmail/client.py`
**セクション**: `class GmailClient`
**行**: L45-L60
**指摘**: Gmail API がレート制限エラーを返した場合のテストがありません
**提案**: `RateLimitError` を mock して retry 処理が動作することを確認するテストを追加してください
```

### 複数箇所をまたがる整合性指摘(design モードで頻出)

```
#### 🔴 A-3(decision)の流れ先が 3 箇所で異なる事実
**対象箇所**(複数):
- `services.md` セクション 3「Queue 構成表」(L82, L83)
- `services.md` シーケンス図 1(L36-49)
- `services.md` シーケンス図 2(L54-72)
**指摘**: 同一事象「A-3(decision)が後続ステップに渡る経路」が、表 / 第 1 図 / 第 2 図で 3 通りに記述されている。実装時に必ず齟齬する
**提案**: 真の方針を 1 つ確定し、3 箇所を一致させる。推奨は「A-3 decision → calendar_queue + notify_queue を **両方 enqueue**」、第 2 シーケンス図の `W->>W` も `W->>Qcal` に修正
```

指摘がない視点は「✅ 問題なし」と記載します。

## 再レビュー(`--since`)モード

修正反映後の差分のみを再レビューするためのモード。「指摘 → 修正 → 再レビュー」のサイクルを効率化します。

### 動作

1. 引数 `--since=<comment-id>` から **起点となる前回レビューコメント** を取得
2. 前回レビュー時点のコミット SHA と HEAD の **差分 commit 一覧** を取得
3. 前回コメントの **全指摘を一覧化**
4. 各指摘について **解決状況を判定**:
   - ✅ 解決
   - ⚠️ 部分解決
   - ❌ 未解決
   - ❓ 確認不能(該当箇所が大幅に変わって直接比較できない)
5. **新規導入された問題** がないかも併せて確認
6. 総合評価で「前回 Critical N 件 → 今回 0 件」のような **解決ステータスサマリ** を提示

### 出力例

```
### 🟢 修正確認

#### 🔴 Critical(3 件 / 全解決)

| ID | 検証結果 |
|----|---------|
| **C1** A-3 流れ先 3 説 | ✅ 完全解決。Queue 表 + 2 シーケンス図が一致 |
| **C2** F-15 ハルシネーション | ✅ F-13 に修正済 |
| **C3** A-10 設計片手落ち | ✅ 5 ファイル横断で完成 |

## 総合評価
判定: ✅ Approve
サマリー: 🔴 0 / 🟡 0 / 🟢 0 / 💡 2
```

## カスタマイズ

`.claude/commands/review.md` を編集することでレビュー観点を調整できます。

**例: プロジェクト固有の観点を追加する場合**

```markdown
## 視点 6: 🗓️ MC 業務固有視点

**確認項目**:
- [ ] Gmail 送信処理に誤送信防止の確認ステップがあるか
- [ ] 司会スクリプトの文言に問題がないか
```

新しい観点はモード(code / design)のいずれかに紐付けるか、新モードを追加してください。
