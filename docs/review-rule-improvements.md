# レビュールール改善点メモ — Application Design レビュー実施で得られた知見

**作成日**: 2026-05-03
**レビュー対象**: PR #3 コメント `4363411004`(Application Design 完了)で示された 5 ファイル
**ベースルール**: `docs/review-guide.md` / `.claude/commands/review.md` / `.github/pull_request_template.md`

レビュー実施を通じて判明した、現行ルールの **設計ドキュメント(非コード成果物)に対する不十分さ** をまとめる。

---

## 1. ルールが「コード PR」前提で設計ドキュメント PR に適合しない

### 現状
- `/review` コマンドは `git diff main...HEAD` を前提とした 5 視点(PO / QA / 開発者 / セキュリティ / 依存)で構成されている
- 「QA 視点」「依存・保守視点」「セキュリティ視点(シークレットハードコード等)」の確認項目は設計ドキュメントに直接当てはめられない
- AIDLC ワークフローでは Inception フェーズで生成される `requirements.md` / `stories.md` / `personas.md` / `application-design.md` 等の **設計ドキュメントだけが変更される PR** が頻繁に発生する

### 提案
- `/review` を **モード化** する: `/review code`(現行) / `/review design`(設計ドキュメント用) / `/review`(自動判定)
- 設計ドキュメント用視点案:
  1. **要件整合性視点**: 上流成果物(requirements / stories / personas)との一貫性
  2. **設計コヒーレンス視点**: 5 ファイル間/章節間の内部矛盾の有無
  3. **完全性視点**: 列挙した要素(コンポーネント / Story / Use Case 等)が全成果物で漏れなく扱われているか
  4. **将来拡張視点**: Phase 2/3 で必要になる拡張ポイントが現設計で阻害されないか
  5. **ハルシネーション視点(設計版)**: 存在しない Story ID / 存在しない型 / 数の合わない集計 等

---

## 2. 「ハルシネーション検出」が code-level に閉じている

### 現状
レビューガイドでは `### 🧪 品質保証 (QA) 視点` の中で「AIハルシネーション検出: 『それらしいが間違った』ロジックがないか」とあるが、対象は実装コードを想定。

### 設計ドキュメントで実際に頻発するハルシネーション
今回のレビューで実際に検出した例:
- 存在しない Story ID への参照(`F-15` を引用しているが実在 Story は `F-14` まで)
- コンポーネント数の不一致(冒頭「約 25 コンポーネント」と表合計「51」)
- 同じ事象を 2 箇所で異なる事実として記述(A-3 decision の流れ先が `notify_queue` / `calendar_queue` で二分)
- 完了基準にあるユースケース数(10)と詳細記述があるユースケース数(9)の不一致(A-10 のメソッド・依存・シーケンスが空白)

### 提案
レビューガイドに **「設計ドキュメントのハルシネーションパターン」** を追加:
- 数値整合性(冒頭サマリ ↔ 表合計 ↔ 各セクション)
- ID 参照整合性(F-XX / U-XX / Story ID 等が上流成果物に実在すること)
- 重複記述の事実一致(同じ事象を 2 箇所以上で説明している場合の整合性)
- 列挙の網羅性(完了基準で挙げた全要素が本文に存在するか)

---

## 3. 上流成果物との整合性チェックが視点として独立していない

### 現状
PO 視点に「実装が要件・Issue の意図を正確に実現しているか」とあるが、これは `要件 → 実装` の検証で、AIDLC のように **要件 → User Stories → Application Design → Construction** と多段階で詳細化していく場合の **段階間整合性** は明示されていない。

### 今回検出した事例
- `requirements.md` の MVP スコープに F8(請求 CSV)が含まれているが、`stories.md` では Phase 2 に移動済み(P2-08)。`application-design.md` も F8 を Phase 2 扱いとしている → **`requirements.md` だけが古い**
- 上流成果物のどれが正かが暗黙化

### 提案
- 「**段階間整合性視点**」を独立した視点として追加(または PO 視点の確認項目として明記)
- AIDLC の特性として「下流ステージで判断が変わったら上流ドキュメントも更新する」ルールをレビュールールに含める
- 各ステージ完了時に「上流ドキュメントへの逆波及が必要か」をチェック項目化

---

## 4. PR テンプレート準拠チェックが AIDLC の段階的 PR と相性が悪い

### 現状
PR テンプレートには「動作確認: ローカルで動作確認済み」のチェックボックスがあり、未チェックは 🔴 Critical 扱い。

### 問題
- AIDLC の Inception フェーズ PR は **設計ドキュメントしか含まない** ため「ローカルで動作確認」は本来不可能
- 「ドキュメントを更新した(必要な場合)」だけが該当するが、メイン作業がドキュメント追加であるため文意が逆
- PR 説明欄は AIDLC ワークフローの最初の commit 時点で書かれ、その後 PR は **ライブの作業ログ的に積み上がっていく**(コメントに進捗が逐次追加される)。最終的なレビュー時点の PR 本文がテンプレートに不備があっても、PR コメント群に十分情報がある

### 提案
- PR テンプレートに **「ドキュメント PR」セクション** を追加するか、テンプレートを **PR 種別(Code / Docs / AIDLC-stage)** で分岐
- AIDLC-stage PR の場合は「動作確認」チェック必須を緩和し、代わりに **「上流成果物との整合性確認」** を必須化
- 事前確認の Critical 判定基準を「PR 種別ごと」に変える

---

## 5. レビュー出力フォーマットがファイル+行番号前提

### 現状
出力フォーマット例:
```
**ファイル**: `path/to/file` (L行番号)
```

### 問題
- 設計ドキュメントの指摘は「セクション 2.3」「`### Story F-12` 配下」「Mermaid 図中の Q5 ノード」など **構造的な参照** が必要
- 1 行内の問題(コードのバグ)ではなく **複数箇所をまたがる整合性問題** が多い

### 提案
- 出力フォーマットに **`セクション` フィールド** を追加(例: `### 5.2 失敗復旧フロー`)
- 整合性問題用に **「対象箇所(複数可)」** フィールドを追加(例: `services.md L82(notify_queue 行) ↔ services.md シーケンス図2 W->>W: A-7`)

---

## 6. 「依存・保守視点」が Markdown 成果物には機能しない

### 現状
依存・保守視点は `package-lock.json` 更新、`npm/pip audit`、新規パッケージ追加の妥当性等。

### 設計ドキュメントでの代替視点
- **後続ステージ(Construction / Functional Design)が現設計で実装可能か**
- **下流ステージで詰まりそうな未確定事項のリスト化**(例: Cron スケジュール頻度、Queue リトライ回数、タイムアウト値)
- **拡張性ポート(F-11 抽象化レイヤ)が将来の仕様変更を本当に吸収できるか**

### 提案
設計レビュー時の依存・保守視点を **「構築可能性視点」** として再定義する。

---

## 7. 重要度レベル(🔴/🟡/🟢/💡)の判定基準が曖昧

### 現状
レベル定義は文章的だが、判定基準は明示されていない。設計ドキュメントの場合「内部矛盾」が後続ステージで実装ミスを生むため Critical 扱いだが、レビュアーによってブレやすい。

### 提案
設計ドキュメント向けに **判定基準のマトリクス** を提供:

| 種類 | 🔴 Critical 例 | 🟡 Warning 例 | 🟢 Suggestion 例 |
|------|--------------|--------------|------------------|
| 内部矛盾 | 同じ事象が 2 箇所で別事実 | 表記揺れ・粒度不一致 | 補足が望ましい |
| ID 参照 | 存在しない Story / 型 を参照 | リンク先未生成 | 別表現が好ましい |
| 数値整合性 | サマリ数 ≠ 詳細合計 | 概算と精密値の混在 | 単位明示の不足 |
| 上流不整合 | 要件と矛盾する設計 | 要件記述が陳腐化 | 用語のゆらぎ |
| 完全性 | 必須要素の欠落(API/UseCase) | 詳細未記載 | 図示があると親切 |

---

## 8. レビュー結果の「フォローアップ確認」プロセスが定義されていない

### 現状
レビュー結果(指摘)→ 修正後の再確認方法が `/review` コマンドからは独立して扱われる。レビュー実施者が再 `/review` を打つしかない。

### 提案
- レビューコメント末尾に **「修正反映確認用 ID」** を付与し、修正後の PR で `/review --since=<id>` で前回からの差分のみ再確認できるようにする
- AIDLC 作業の特性上、**「指摘 → 修正 → 再レビュー」を多サイクル回す前提のフロー** をルール化
