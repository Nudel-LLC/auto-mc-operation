## 2. 業務背景・課題

### 2.1 業界構造
- MC/コンパニオンは平均 6〜7 社の事務所に所属
- 各事務所はメーリングリストで案件募集メールを所属 MC 宛に配信
- 事務所ごとにメール書式・フォーマットが**統一規格なし**
- MC は個別にメール返信してエントリー、決定後は重複案件の辞退連絡が必要

### 2.2 現状の業務フロー(手動)
1. 各事務所からの案件募集メールを受信(1日複数通)
2. メール内容(日程・場所・報酬等)を確認
3. 自分のカレンダーで日程の空きを確認
4. 空いていればメール返信でエントリー希望、自己 PR を添付
5. カレンダーに「仮」予定を手動登録
6. 決定後はカレンダーを「確定」に更新
7. 重複する仮案件の辞退メールを各事務所に個別送信
8. 辞退案件をカレンダーから削除

### 2.3 業務上の課題
- 複数事務所からのメール管理が煩雑
- 事務所ごとに書式が異なり情報読み取りに時間がかかる
- カレンダー手動反映の漏れ
- 部分重複を含む日程重複の判定が複雑
- 辞退連絡の送信漏れ・遅延
- エントリー時の PR 文面作成が面倒
- ウェットなコミュニケーションコストが全体的に高い

## 3. ユーザーペルソナ

### 3.1 プライマリユーザー(MVP 対象)
- **職業**: MC / コンパニオン
- **所属事務所数**: 平均 6〜7 社
- **使用ツール**: Gmail / Google カレンダー / LINE
- **MVP 段階の規模**: 1 名(開発者本人または知人)
- **課題**: 大量の案件メール管理と関連業務(カレンダー管理・辞退連絡)

### 3.2 拡張時のユーザー(Phase 2 以降)
- 同業の MC・コンパニオン全般(マルチテナント SaaS として展開)
- 上限想定: **5,000 ユーザー × 20件/日 = 100,000 件/日**

### 3.3 将来の拡張ユーザー(Phase 3 以降)
- Gmail 以外のメールサービス利用者
- Google カレンダー以外のスケジュールサービス利用者

## 4. プロダクトスコープ

### 4.1 MVP スコープ(Phase 1)
利用ユーザーは **1 名(開発者本人または知人)**。以下の機能を含むエンドツーエンドパイプラインを提供する。

| ID | 機能 | 概要 |
|----|------|------|
| **F1** | Gmail メール自動分類 | 受信メールを「案件募集 / 決定連絡 / その他」に分類 |
| **F2** | 案件情報抽出 | 日程・場所・報酬・事務所名・締切日時・案件名・PR要素・その他条件を構造化抽出 |
| **F3** | カレンダー空き確認 | Google カレンダーと突き合わせて部分重複・移動時間を考慮した重複判定 |
| **F4** | エントリー下書き自動作成 | 過去エントリーメールから学習した PR 文を含む返信下書きを Gmail 下書きフォルダに保存 |
| **F5** | LINE 通知 | 処理結果(分類・抽出・空き確認・下書き作成・辞退候補)を LINE Messaging API でリアルタイム通知 |
| **F6** | カレンダー自動管理 | エントリー時に「仮」予定登録、決定時に「確定」更新、辞退時に削除 |
| **F7** | 辞退連絡(半自動) | 重複検出時に辞退メール下書きを作成し、ユーザー承認後に送信 |
| **F8** | 請求 CSV エクスポート | 月次・事務所別の確定案件を CSV 出力(月・事務所名・案件名・日程・場所・報酬・ステータス) |

### 4.2 MVP に含めないもの(Phase 2 以降)
- マルチテナント機能(認証・テナント分離・課金)
- ユーザー操作 UI(Web ダッシュボード・スマホアプリ)
- ユーザー手動による実績登録機能(PR 文学習データの追加入力)
- Gmail 以外のメールサービス対応
- Google カレンダー以外のカレンダー対応
- エントリー送信の完全自動化(MVP は下書き保存に留める)

### 4.3 自動化レベル
- **エントリー送信**: 下書きのみ生成し、Gmail 下書きフォルダに保存(送信はユーザーが手動)
- **辞退送信**: 半自動(下書き生成→ユーザー承認後送信)
- **カレンダー反映**: 自動(仮予定登録/確定更新/削除)

## 5. 機能要件

### FR-1: Gmail メール自動分類
- **入力**: Gmail Push 通知 (Pub/Sub Watch) で受信した新着メール
- **出力**: 分類ラベル(`recruitment` / `decision` / `other`)
- **手段**: Claude Haiku で判定
- **要件**:
  - 分類精度の評価指標を定義する(F1 score 等)
  - 誤分類は監査ログに記録
  - 信頼度が閾値未満の場合は `other` にフォールバック

### FR-2: 案件情報抽出
- **入力**: 分類が `recruitment` のメール本文・件名・送信元
- **出力**: 構造化データ(JSON)
- **抽出項目**:
  | 項目 | 型 | 必須 |
  |------|----|----|
  | `office_name`(事務所名) | string | Yes |
  | `subject_name`(案件名) | string | Yes |
  | `start_at` / `end_at`(日程) | datetime | Yes |
  | `location`(場所) | string | Yes |
  | `compensation`(報酬) | string + 構造化(金額・支払条件) | Yes |
  | `deadline_at`(エントリー締切) | datetime | No(あれば) |
  | `pr_required`(PR 文必要かどうか) | boolean | Yes |
  | `other_conditions`(その他条件) | string | No |
- **要件**:
  - 抽出失敗項目は `null` を許容しつつ、必須項目欠落は LINE で警告通知
  - 事務所ごとの書式パターンを学習・蓄積

### FR-3: カレンダー空き確認・重複判定
- **入力**: 抽出済みの日程
- **要件**:
  - **部分重複 + 移動時間バッファ(前後 1 時間)** を考慮した重複判定
  - Google Calendar API でユーザー本人のカレンダーを参照
  - 仮予定 / 確定予定 / 他カレンダー(プライベート) を区別

### FR-4: エントリー下書き自動作成
- **入力**: 抽出データ + 過去エントリーメール(自動学習データ)
- **出力**: Gmail 下書きフォルダに保存された返信下書き
- **要件**:
  - **PR 文の元データ**: 過去ユーザーが書いたエントリーメールから自動学習(手動登録 UI は MVP では非対応)
  - PR 要素が必要(`pr_required = true`)な場合のみ PR 文を含める
  - 下書きには件名・本文・送信先(In-Reply-To 維持)を含める

### FR-5: LINE 通知
- **手段**: LINE Messaging API(公式 Bot)
- **通知タイミング**:
  - 案件募集を検出・分類完了した時
  - 抽出に必須項目欠落があった時
  - 重複検出時(辞退候補)
  - エントリー下書き作成完了時
  - 決定連絡を検出した時
- **メッセージ構成**: 案件サマリ(事務所・日程・場所・報酬)+ アクションボタン(承認/却下)

### FR-6: カレンダー自動管理
- **要件**:
  - エントリー(下書き作成)時: Google カレンダーに「仮」予定登録(タイトル接頭辞 `[仮]`、色分け)
  - 決定連絡を検出時: 「仮」→「確定」に更新(`[仮]` プレフィックス削除)
  - 辞退時(承認後): カレンダーから該当予定を削除

### FR-7: 辞退連絡(半自動)
- **要件**:
  - 決定案件と日程重複(部分含む)する仮案件を検出
  - 各事務所宛の辞退メール下書きを生成し、LINE で承認依頼
  - ユーザー承認後に Gmail API で送信

### FR-8: 請求 CSV エクスポート
- **要件**:
  - 月次・事務所別に確定案件を出力
  - 列: 月 / 事務所名 / 案件名 / 日程(開始・終了)/ 場所 / 報酬 / ステータス(確定/辞退/未決定)
  - 文字コード: UTF-8 BOM 付き(Excel 対応)
  - ダウンロード手段は MVP では LINE 経由(LINE で要求 → 一時 URL を返す)

## 6. 非機能要件

### NFR-1: 性能・レイテンシ
- **メール受信から LINE 通知までの目標時間**: **5 分以内**
- **同時処理**: Cloudflare Queues による非同期処理で詰まりを許容
- **タイムアウト**: 各 LLM 呼び出し 25 秒、外部 API 呼び出し 30 秒

### NFR-2: スケーラビリティ
- **MVP**: 1 ユーザー × 〜20 件/日(600 件/月)
- **目標スケール**: **5,000 ユーザー × 20 件/日 = 100,000 件/日**(大規模 SaaS)
- **アーキテクチャ要件**: Cloudflare Workers + Queues + D1 で水平スケール、拡張時に Cloud Run + Pub/Sub + Cloud SQL に移行

### NFR-3: 可用性
- **MVP**: 99.0% (個人運用許容範囲)
- **拡張時**: 99.5% 以上(Cloudflare の SLA に準拠)

### NFR-4: セキュリティ(Security Baseline 拡張 = 強制)
- **SECURITY-01**: D1 / KV / R2 すべて暗号化(Cloudflare 提供のデフォルト暗号化)、TLS 1.2+ 強制
- **SECURITY-02**: Cloudflare Workers のアクセスログを Logpush で R2 / 外部 SIEM に送出
- **SECURITY-03**: 構造化ロギング(timestamp / request_id / level / message)、PII / トークンを含めない
- **SECURITY-04**: HTTP セキュリティヘッダ(CSP / HSTS / X-Content-Type-Options / X-Frame-Options / Referrer-Policy)を全レスポンスに付与
- **SECURITY-05**: 全 API エンドポイントで入力検証(Rust の `serde` + `validator` クレート、長さ・型・正規表現チェック、SQL は `sqlx` のパラメータバインドのみ)
- **SECURITY-06**: Cloudflare API トークン・GCP IAM は最小権限・リソース限定
- **SECURITY-08**: アプリレベル認可(JWT / セッション)、CORS は明示オリジンのみ
- **SECURITY-09**: 本番エラーレスポンスにスタックトレース等を含めない、デフォルト認証情報排除
- **SECURITY-10**: `Cargo.lock` コミット、`cargo audit` を CI に組込み、SBOM 生成、Docker `latest` 禁止
- **SECURITY-11**: 設計フェーズで脅威モデリング(STRIDE)を実施

### NFR-5: プライバシー・データ保存(個情法・電気通信事業法対応)
- **保存方針(ハイブリッド)**:
  | データ種別 | 保存期間 | 保管先 |
  |-----------|---------|--------|
  | メール本文(原文) | **30 日**(以降は自動削除) | R2(暗号化) |
  | 構造化抽出データ | **24 ヶ月** | D1 → Cloud SQL |
  | PR 文学習データ(本人作成エントリー文) | **永続**(本人帰属) | D1 → Cloud SQL |
  | 監査ログ・処理メタデータ | **12 ヶ月** | R2(Logpush) |
- **法的整理**:
  - 電気通信事業法 4 条「通信の秘密」: 本サービスは本人のメール処理代行のため電気通信事業者非該当が一般的整理(ユーザー同意必須)
  - 個情法: 利用目的特定・通知、安全管理措置、第三者提供禁止(LLM は業務委託整理)、Anthropic 米国保管に伴う海外移転同意
- **同意管理**: 初回セットアップ時にプライバシーポリシー同意を取得・保管

### NFR-6: 信頼性・テスト(Property-Based Testing 拡張 = 強制)
- **PBT 対象**:
  - 重複判定ロジック(時間範囲の重なり計算)
  - メール分類器(入力サニタイズ・出力スキーマ往復)
  - 構造化抽出のシリアライズ往復(JSON ↔ Rust 型)
  - CSV エクスポートのエンコード往復
- **手段**: Rust の `proptest` クレート
- **ユニットテストカバレッジ目標**: 80% 以上(ビジネスロジック層)

### NFR-7: コスト
- **コスト制約**: 品質・速度を優先(コスト上限なし)
- **MVP 段階の目安**: 月額 〜10,000 円程度を想定(Claude Haiku + Cloudflare 従量 + LINE Messaging 無料枠内)

### NFR-8: 監視・運用
- **構造化ログ**: すべてのリクエスト・処理に `request_id` / `user_id` / `correlation_id` を付与
- **監査ログ**: 案件分類結果・エントリー送信履歴・辞退送信履歴・カレンダー操作履歴
- **エラー通知**: 重要なエラー(LLM 失敗・API 失敗・データ整合性エラー)を LINE で本人に通知

## 7. 技術スタック・アーキテクチャ方針

### 7.1 言語・フレームワーク
- **バックエンド言語**: **Rust**
- **Cloudflare Workers ランタイム**: `workers-rs`(WASM 実行)
- **Cloud Run 拡張時**: `axum` + tokio
- **HTTP クライアント**: `reqwest`(REST 直叩き、Anthropic / Google / LINE すべて)
- **OAuth2**: `yup-oauth2`(Google API のみ)
- **シリアライズ**: `serde` + `serde_json`
- **入力検証**: `validator` クレート
- **テスト**: `proptest`(プロパティベース)+ 標準テスト

### 7.2 ホスティング
- **MVP**: Cloudflare(Workers / D1 / KV / R2 / Queues / Cron Triggers / Durable Objects)
- **拡張時**: GCP(Cloud Run / Cloud Scheduler / Cloud SQL / Pub/Sub / Cloud Storage)
- **移行戦略**: D1(SQLite)→ Cloud SQL(PostgreSQL)は SQL 方言差が小さく、`sqlx` のクエリは概ね両立

### 7.3 データ永続化
| データ種別 | MVP (Cloudflare) | 拡張時 (GCP) |
|-----------|------------------|-------------|
| 案件・エントリー・カレンダー・実績 PR(リレーショナル) | **D1 (SQLite at edge)** | **Cloud SQL (PostgreSQL)** |
| セッション・冪等性キー(KV) | **Cloudflare KV** | **Memorystore (Redis)** |
| メール原文・添付・大容量ログ | **Cloudflare R2** | **Cloud Storage** |
| Gmail Watch 状態(per-user 強整合) | **Durable Objects + SQLite** | **Firestore + Cloud Tasks** |
| 非同期キュー | **Cloudflare Queues** | **Pub/Sub + Cloud Tasks** |

### 7.4 外部サービス連携
- **Gmail API**: Push 通知(Pub/Sub Watch)+ Bearer トークン直叩き
- **Google Calendar API**: REST 直叩き(`yup-oauth2` で認証)
- **OAuth スコープ**: 読み取り + 書き込み(送信・編集を含むフルアクセス)
- **LINE Messaging API**: REST 直叩き(署名検証含めて 100〜200 行)
- **Anthropic API**: REST 直叩き(`reqwest` + JSON、Claude Haiku のみ)

### 7.5 UI
- **MVP**: UI なし(LINE 通知 + Gmail 下書きで完結)
- **Phase 2+**: 必要に応じて Web ダッシュボード追加(`leptos` / `yew` 等の Rust SPA、または React)

## 8. 制約・前提

### 8.1 制約
- **Cloudflare Workers の WASM サイズ・CPU 時間制限**: workers-rs のバイナリサイズ最適化が必要
- **Anthropic / Google / LINE の Rust 公式 SDK が無い**: REST 直叩きでの実装(リスク: API 仕様変更の追従コスト)
- **LINE Notify は終了**: LINE Messaging API(公式 Bot)を使用

### 8.2 前提
- ユーザー本人の Google アカウント・LINE アカウントの OAuth 認可が得られている
- ユーザーが事務所からの案件募集メールを受信する Gmail を保有している
- Anthropic API キーが取得可能(品質優先のためコスト制約なし)
