# ペルソナ — auto-mc-operation

このドキュメントは User Stories のペルソナ定義です。
**プライマリペルソナ 2 名** を詳述します(P1: AI に慣れた MC / P2: AI 未経験の MC)。

両ペルソナを念頭に Story を書くことで、AI リテラシーに依らず使えるサービスとして設計できます。

---

## P1: MC / コンパニオン本人(プライマリペルソナ)

### 基本属性
| 項目 | 内容 |
|------|------|
| 名前(仮称) | 佐藤 美咲(さとう みさき) |
| 職業 | フリーランス MC / イベントコンパニオン |
| 年齢層 | 25〜35 歳 |
| 拠点 | 東京・大阪等の都市圏 |
| 所属事務所 | 平均 6〜7 社(個別契約) |
| 月間稼働 | 15〜25 件の案件、うち平均 50% が「打診→エントリー→落選 or 辞退」 |
| 月収レンジ | 案件規模により大きく変動 |

### 利用ツール環境
- **メール**: Gmail(個人アカウント、事務所からの案件募集メーリングリストを集約)
- **カレンダー**: Google カレンダー(複数カレンダーを併用、プライベートと案件を区別)
- **コミュニケーション**: LINE(個人連絡・グループ・公式アカウント)
- **デバイス**: スマートフォン中心(iPhone)、補助的に PC
- **技術リテラシー**: 中(Web サービスの初期セットアップは可能、専門用語は不慣れ)

### ゴール(目的)
1. **エントリー機会を逃さない** — 案件募集メールに気付かず逃すと収入機会喪失
2. **重複案件で信頼を失わない** — 二重ブッキングや辞退漏れは事務所との信頼関係に直結
3. **事務作業時間を減らし、本業(MC スキル磨き・体調管理)に時間を使う**
4. **収入と稼働実績を月次で把握し、確定申告・営業に活用する**

### 主要なペインポイント(現状の課題)
| # | ペイン | 現状の影響 |
|---|--------|-----------|
| PP-1 | 6〜7 社からの案件募集メールが日中バラバラに届く | スマホ通知が多すぎて見逃し・通知疲れ |
| PP-2 | 事務所ごとに書式が違い、日程・場所・報酬の読み取りに時間がかかる | 1 通あたり 5〜10 分の確認作業 × 1 日数通 |
| PP-3 | カレンダーへの仮予定登録を忘れがち | 後から「あの案件入れてたっけ?」と不安 |
| PP-4 | 部分的に重複する日程の判定が頭で計算しにくい | 「移動時間入れたら無理かも」を都度検討 |
| PP-5 | 決定後の辞退連絡を複数事務所に送る作業が手間 + 送信漏れリスク | 信頼低下・最悪契約解除 |
| PP-6 | 自己 PR 文を毎回似た内容で書き起こす | 1 通あたり 5 分、月数十回 |
| PP-7 | 月次の確定案件・収入を集計するのに表計算ソフトで手作業 | 月末に数時間 |

### モチベーション(行動原理)
- **時間より信頼**: 多少時間がかかっても、事務所への失礼がない方を優先する
- **自分でコントロールしたい**: メール送信の最終承認は自分が行いたい(完全自動化への抵抗)
- **スマホで完結したい**: PC を開かずスマホ + LINE で意思決定したい
- **柔軟性**: 案件単位で対応を変えたい(全自動だと困るケースがある)

### 技術的な前提
- Google アカウント・LINE アカウントを保有しており、OAuth 認可手続きが可能
- システムへの初期セットアップ(LINE 公式アカウントを友達登録する等)は可能
- データ取扱いに関する基本的なプライバシーポリシーは読んで同意する

### サービス利用上の期待
- **5 分以内**に案件募集メールの内容を LINE で受け取りたい(NFR-1 対応)
- **半自動承認**で操作したい(完全自動は怖い、Q3 = C / Q4 = B 対応)
- **個人情報を漏らさない**(NFR-5 ハイブリッド保存方針対応)
- **月額 500 円程度**であれば継続したい(NFR-7 価格目標対応)

### サービス導入に対する不安・抵抗
- 「メール内容を AI に渡す」ことへの心理的抵抗(プライバシーポリシーで解消)
- 「勝手に送信される」ことへの抵抗(下書き保存 / 半自動承認で解消)
- 「事務所に AI 利用がバレて評価が下がる」懸念(下書きはユーザー編集可能で解消)

### このペルソナを念頭に Story を書く時のチェックポイント
- ✅ ユーザーは **常にスマホ + LINE** から操作する想定
- ✅ ユーザーは **送信前に必ず確認したい**(下書きで止める / 半自動承認)
- ✅ ユーザーは **複数案件の並行管理** が日常(Story は単発でなく時系列で考える)
- ✅ ユーザーは **API・OAuth 等の用語に詳しくない**(エラーメッセージは平易に)
- ✅ ユーザーは **信頼を最優先**(辞退漏れ・誤送信は致命的)

## P2: MC / コンパニオン本人(AI 未経験者・プライマリペルソナ #2)

### 基本属性
| 項目 | 内容 |
|------|------|
| 名前(仮称) | 中村 ゆかり(なかむら ゆかり) |
| 職業 | フリーランス MC / イベントコンパニオン |
| 年齢層 | **30 歳前後** |
| 拠点 | 都市圏(東京・大阪等) |
| 所属事務所 | 平均 6〜7 社 |
| 月間稼働 | 15〜25 件、傾向は P1 と同じ |
| 月収レンジ | P1 と同じ |

### 利用ツール環境
- **メール**: Gmail(普段使い、スマホで確認)
- **カレンダー**: Google カレンダー(基本的なイベント登録のみ、共有や複数カレンダーには不慣れ)
- **コミュニケーション**: LINE(日常生活・仕事連絡の主要ツール)
- **デバイス**: スマートフォン中心(Android / iPhone どちらも)、PC は最低限のみ
- **技術リテラシー**: **低〜中**(Web サービスの基本操作はできるが、専門用語に強い苦手意識)
- **AI 経験**: **ゼロ**(ChatGPT / Claude / Gemini / Copilot などを **一度も触ったことがない**)

### ゴール(目的)
1. **エントリー機会を逃さない**(P1 と共通)
2. **重複案件で信頼を失わない**(P1 と共通)
3. **事務作業時間を減らし本業に集中したい**(P1 と共通)
4. **「複雑なツール」を覚えなくても使えること**(← P2 固有)

### 主要なペインポイント(P1 との差分を中心に)
| # | ペイン | 現状の影響 / P2 固有の側面 |
|---|--------|---------------------------|
| PP-1 | 6〜7 社からのメール管理 | P1 と共通 |
| PP-2 | 事務所書式の違い | P1 と共通 |
| PP-3 | カレンダー反映漏れ | P1 と共通 |
| PP-4 | 部分重複の判断 | P1 と共通 |
| PP-5 | 辞退連絡の手間 | P1 と共通 |
| PP-6 | PR 文の毎回作成 | P1 と共通 |
| PP-7 | 月次集計の手作業 | P1 と共通 |
| **PP-A**(P2 固有) | **「AI を使うサービス」と聞くだけで身構える** | サービス導入の心理的ハードルが高い |
| **PP-B**(P2 固有) | **「OAuth」「API」「トークン」等の用語が出ると混乱** | セットアップ時に挫折するリスク |
| **PP-C**(P2 固有) | **エラーメッセージが英語混じり / 技術用語混じりだと意味がわからない** | 失敗時に何をすればいいかわからず放置 |
| **PP-D**(P2 固有) | **ステップが多いと最後まで完了できない** | 初期セットアップで離脱 |

### モチベーション(行動原理)
- **複雑なものを覚えたくない**: スマホでポチポチするだけで完結してほしい
- **失敗が怖い**: 「変なボタン押して取り返しがつかなくなる」のが最も嫌
- **わかりやすい言葉**: 「OAuth 認可」より「Google と連携」、「LLM」より「AI で自動分析」
- **手厚いガイド**: 失敗時に「これをタップして」という具体的な誘導が欲しい
- **同じ業界の人が使っている安心感**: 評判 / クチコミを重視

### 技術的な前提
- Google アカウント・LINE アカウントは **既に持っているが、設定変更などはしたことがない**
- LINE 公式アカウントを「友だち追加」する操作はできる
- スマホで URL を開いてボタンをタップすることはできる
- 細かいプライバシーポリシーの文言を全部読む習慣はない(が、要点は知りたい)

### サービス利用上の期待(P1 との差分)
- **セットアップは LINE 内で完結**(別タブで難しい画面に行く回数を最小化)
- **専門用語ゼロ**(エラーメッセージも含む)
- **失敗時の次のアクションが具体的に書かれている**(「再認可してください」ではなく「LINE で『再連携』とタップしてください」)
- **AI を使っているという自覚がなくても良い**(「Bot が代わりに整理してくれる」程度の理解で十分)

### サービス導入に対する不安・抵抗(P1 との差分)
- **AI に対する漠然とした不安**(「勝手に何かされるのでは」「データが流出するのでは」)
- **専門用語が出てきた瞬間に離脱する** リスクが高い
- **トラブル時に問い合わせる窓口がないと不安**

### このペルソナを念頭に Story を書く時の **追加** チェックポイント
- ✅ ユーザー向けメッセージ(LINE 通知 / エラー / 同意画面)は **小学校高学年が読める日本語**
- ✅ 専門用語(OAuth, API, トークン, 認可, スコープ など)は **ユーザー向けには使わない**
- ✅ 失敗時の通知には **「次に何をタップすればいいか」を具体的に明記**
- ✅ セットアップは **LINE トーク内で完結する誘導** を最大化(外部 URL は最小限)
- ✅ プライバシーポリシーは **3 行のサマリ + 詳細リンク** の形で提示

## P1 / P2 比較表(共通点・差分)

| 観点 | P1: AI に慣れた MC | P2: AI 未経験の MC |
|------|------------------|------------------|
| 業界・所属事務所数 | 同じ | 同じ |
| ゴール(機能面) | 同じ | 同じ |
| 業界由来のペインポイント(PP-1〜7) | 同じ | 同じ |
| **AI リテラシー** | 中(ChatGPT 等を触ったことがある) | **ゼロ** |
| **専門用語耐性** | 中 | 低 |
| **セットアップ完遂率** | 高 | 低(脱落リスク) |
| **失敗時の自己解決力** | 中 | 低(具体ガイド必須) |
| **AI 導入への心理的抵抗** | 低 | 中〜高 |
| **適切な UX** | 効率重視・最短導線 | **わかりやすさ重視・手厚いガイド** |

### 設計上の含意
- 機能要件(F1〜F8)・非機能要件(NFR-1〜NFR-8)は **両ペルソナ共通**
- ただし **ユーザー向けメッセージング(LINE 通知 / 同意画面 / エラー文言)は P2 を最低基準** として設計
- これにより P1 にとっても「丁寧でわかりやすい」サービスになり、不利益はない

## 周辺の関係者(ペルソナ非対象、参考)

Q1 = A の方針に従い詳細ペルソナは作成しないが、Story 内で関係者として登場するため、最低限の概念定義をしておく。

- **事務所担当者(Office Staff)**: メールの送信元 / 辞退メールの宛先。本サービスのユーザーではない
- **サービス運営者(Operator)**: 開発者本人 / 拡張時の SaaS 提供者。Anthropic API キーを保有・管理。本サービスのエンドユーザーストーリーには含まれない(運用者ストーリーは Phase 2 以降)
